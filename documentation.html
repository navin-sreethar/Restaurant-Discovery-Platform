RestaurantIQ

Technical Documentation & Project Specifications

1. Setup Instructions

The entire application is containerized using Docker for a seamless setup experience. To run the application:

# 1. Clone the repository
git clone <repo-url>
cd Restaurant-Discovery-Platform

# 2. Build and start all services
docker-compose up -d --build

# 3. View the application
# Frontend: http://localhost:5173
# Backend API: http://localhost:8000

4. API Documentation (Swagger)

Because the backend is built with FastAPI, comprehensive OpenAPI documentation is generated automatically. The platform does not require external Postman collections as it features an interactive Swagger UI directly in the browser.

  • Swagger UI (Interactive API Explorer): http://localhost:8000/docs
  • ReDoc (Alternative format): http://localhost:8000/redoc
  • Raw OpenAPI JSON: http://localhost:8000/openapi.json

Note: The backend Docker container must be running to view the live documentation.

5. Assumptions Made

  • Authentication Flow: We assume a private, invite-only B2B system where new registrations default to is_approved = False. Users cannot interact with the platform until an Administrator explicitly approves their account in the dashboard.
  • AI Guardrails: It is assumed that API keys for Google Gemini will be injected via .env at runtime. The system expects standard inputs, and uses comprehensive `try/except` blocks to gracefully handle potential Gemini API safety filters or timeouts without crashing.
  • Data Scalability: It is assumed that the database could scale significantly in an enterprise environment. As such, explicit database indexing is applied to common filter fields (city, cuisine, name) and server-side pagination (12 items per page) is strictly enforced on the main feed to reduce memory overhead.
  • Deployment Strategy: We assume a containerized production environment. The use of docker-compose ensures identical execution across development, staging, and production environments, eliminating "it works on my machine" issues.
  • Logging: Python's built-in logging module writes to standard output, which is the assumption for Dockerized applications where log streams are aggregated and monitored by the container orchestrator.
